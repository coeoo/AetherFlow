## 🎯 项目背景

安全团队面对的核心问题不是“缺少一个页面抓取脚本”，而是缺少一套能够把分散
信号转成可解释结果的智能处理系统。

对于 CVE 补丁搜索来说，真实世界中的补丁路径经常并不直接：

- CVE 记录可能只给出公告页或漏洞跟踪页
- 安全公告可能还要继续跳转到 vendor tracker、Bugzilla、代码评审页
- 最终 patch 可能藏在 attachment、PR patch、MR patch、commit patch 或
  其他下载入口中

因此，AetherFlow 的目标不是继续堆叠关键词匹配，而是构建一个
**受控、多跳、可审计的 Patch Agent**。

---

## 💡 产品定位

AetherFlow 是一个以图运行时为核心的安全情报处理引擎。

平台层提供统一任务调度、文档抓取、Artifact 留存、证据表达与结果投递；
场景层在平台能力之上实现具体智能体主链。

首批场景包括：

- **CVE Patch Agent**
  输入一个 CVE，主动聚合多源 seed，受控探索相关页面，
  在有限预算内找到可信 patch，并保留完整搜索图与证据链。
- **安全公告提取**
  输入或监控公告源，抽取结构化情报、证据片段和处置建议，
  并接入投递链路。

## 👥 目标用户

### 主要用户群体

- **安全分析师**：需要快速拿到补丁线索、证据路径和可复核结论。
- **漏洞响应团队**：需要在漏洞暴露后尽快收敛到可执行的修复入口。
- **威胁情报工程师**：需要可追溯的抓取证据、搜索路径和结果结构化输出。
- **SecOps 团队**：需要把分析结果接入邮件、企业微信或 Webhook 工作流。

### 用户痛点

1. 安全信号分散在多个来源站点和页面层级中，人工跳转成本高。
2. 传统规则链通常只能命中显式 patch，无法覆盖 bridge page 和多跳链路。
3. 即便拿到结果，往往也缺少搜索路径、证据来源和可复查的原始内容。
4. 分析结果与投递动作脱节，难以进入团队日常流程。

## ✨ 核心价值

### 对用户的价值

- 从“输入 CVE”直接走到“可信 patch 与证据”，减少人工搜索链路。
- 让结果具备解释性，用户可以复查系统为什么沿某条路径继续深入。
- 未命中 patch 时也能得到完整搜索图、停止原因和后续建议。
- 把一次性分析、持续监控和结果投递收敛到同一个承载平台。

### 对产品的价值

- 把多跳网页探索、候选收敛和证据表达沉淀成平台级能力，而不是零散脚本。
- 用真实的 Patch Agent 主链验证 LangGraph、预算控制和审计模型的边界。
- 为更多安全情报场景复用统一底座，而不是重复建设采集与投递体系。

---

## 🧱 产品结构

产品采用“平台底座 + 场景智能体”的结构：

1. **平台底座**
   统一任务、调度、抓取、Artifact、投递、状态页与首页摘要。
2. **Patch Agent 场景**
   围绕 `run + search_graph + decision_history + candidate_convergence`
   组织搜索与结果表达。
3. **安全公告场景**
   围绕公告源管理、批次运行、情报包和投递闭环组织。

## 🗓️ 里程碑方向

| 阶段 | 目标 |
|------|------|
| 文档与设计 | 统一产品叙事、Agent 主线、数据模型与页面规格 |
| 平台底座 | 提供任务、Artifact、抓取、投递与系统状态共用能力 |
| Patch Agent 主链 | 完成多源 seed、受控多跳搜索、预算控制与搜索图回放 |
| 公告场景闭环 | 完成手动提取、监控批次、情报包详情与投递闭环 |
| 联调发布 | 完成前后端联调、验证与首版对外交付 |

---

## 📝 边界说明

- 产品主线是“受控多跳智能体”，不是开放式通用网页代理。
- 模型负责搜索决策，但必须运行在规则护栏、预算上限和证据审计之内。
- 首页、工作台和详情页都服务于“先给结论，再给搜索图与证据”的体验。
