# 代码规范

> **AetherFlow 项目代码规范**

---

## 1. 总体原则

1. 平台能力与场景能力必须分层。
2. 真实闭环优先于抽象。
3. 可测试性优先于“写起来顺手”。
4. 不为兼容旧结构而保留新仓无意义的 legacy 代码。

---

## 2. 目录与边界规范

### 平台层
- 统一放平台共用能力：任务、投递、Artifact、抓取、健康检查。
- 不允许出现 CVE 或公告语义命名。

### 场景层
- `cve` 和 `announcement` 各自维护自己的模型、服务、路由和结果类型。
- 不允许用一个“通用 run/result 类”硬合并两个场景。

---

## 3. 命名规范

- 变量、函数、类名必须清晰表达含义。
- 禁止使用无意义缩写，如 `tmp`、`data2`、`flag`。
- 平台层命名不带场景词。
- 场景层命名必须明确场景归属，例如 `CVERun`、`AnnouncementRun`。

---

## 4. 注释规范

- 复杂业务逻辑、状态流转、抓取边界要写简体中文注释。
- 注释解释“为什么这样做”，而不是重复代码字面含义。
- 不保留大段被注释掉的旧实现。

---

## 5. 错误处理

- 快速失败，错误信息带上下文。
- 不静默吞异常。
- 外部抓取、模型调用、投递失败都必须记录可排查信息。

---

## 6. 数据与接口规范

- 平台 API 与场景 API 分开命名和分目录。
- 大文本、HTML、patch 等内容统一走 Artifact，不直接堆进业务表响应。
- 任何结构化结果都要有证据片段或来源引用。

---

## 7. 测试规范

- 新功能先写设计，再写测试，再写实现。
- 长任务测试要覆盖成功、失败、重试、超时四类路径。
- 场景测试必须覆盖手动入口与自动入口差异。
- 测试命令设置超时，避免后台任务卡死。

---

## 8. 前端规范

- 首页必须围绕“场景入口”设计，不能回到后台菜单壳。
- 运行态页面优先展示结论和关键信息，不先暴露原始 JSON。
- 异步状态统一处理 loading、error、empty、success 四类状态。

---

## 9. 文档规范

- 功能设计优先更新 `04-功能设计/`
- 数据库变更必须同步更新 `03-系统架构/数据库设计.md` 和 `05-数据库/sql/`
- 每次会话结束必须补 AI 开发日志
