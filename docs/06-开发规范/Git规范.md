# Git 规范

> **AetherFlow Git 协作约定**

---

## 1. 基本原则

1. 文档优先，功能设计确认后再进入实现。
2. 每次提交要说明“改了什么、为什么改、验证了什么”。
3. 不用模糊提交信息，如 `fix bug`、`update`。

---

## 2. 提交信息格式

```text
<type>(<scope>): <subject>
```

示例：

```text
docs(platform): 完成任务与投递中心功能设计
feat(cve): 实现 fast-first run 创建与详情查询
test(announcement): 补充监控源调度回归测试
```

---

## 3. 常用 type

- `docs`：文档更新
- `feat`：新增功能
- `fix`：修复问题
- `refactor`：重构
- `test`：测试补充
- `chore`：杂项维护

---

## 4. 提交前检查

- [ ] 相关设计文档已更新
- [ ] 相关测试已补充或说明未补原因
- [ ] 不包含无关文件
- [ ] 已补 AI 开发日志

---

## 5. 分支与推送

- 默认采用小步提交
- 文档阶段可以先集中为文档提交
- 功能阶段按模块拆分提交，避免一个提交跨太多模块
