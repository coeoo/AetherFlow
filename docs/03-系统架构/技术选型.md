# 技术选型

> **技术栈选择和理由**

> 读前建议：先阅读 [`总体设计`](../00-总设计/总体项目设计.md)。本文负责把总设计中的技术哲学投影为具体选型；涉及前端实现边界时，再结合 `前端架构设计.md` 阅读。

---

## 🎯 选型原则

1. **真实闭环优先**：优先选择能最快支撑场景闭环的技术。
2. **结构清晰优先**：平台与场景边界必须靠技术结构表达出来。
3. **维护成本可控**：不同时维护多套数据库后端或历史兼容层。
4. **结果可追溯**：技术选型要支撑证据留存、结果复查和任务审计。
5. **对旧行为兼容，不对旧结构兼容**：参考旧项目能力，不继承旧工程结构。

## 💻 前端技术选型

### 技术选择
| 技术 | 选型 | 版本 |
|------|------|------|
| 框架 | React | 18.x |
| UI 层 | 自定义组件 + CSS 变量体系 | 自定义 |
| 状态管理 | TanStack Query + 组件内状态 | 5.x |
| 构建工具 | Vite | 6.x |
| 路由 | React Router | 6.x |

### 选型理由
**选择 React + Vite 的原因**：
- 与参考项目已有前端经验一致，认知切换成本最低。
- 适合构建“平台首页 + 场景工作台 + 详情页”这类路由型应用。
- 配合 TanStack Query 能较好承载长任务轮询与运行态展示。

**为什么不引入重型 UI 框架**：
- 当前产品需要建立自己的平台视觉语言，避免被后台模板绑架。
- 模块交互和信息密度较高，自定义组件更容易服务场景表达。

**前端工程约束**：
- 服务端状态统一由 TanStack Query 管理，不引入全局状态库。
- 页面组件不直接发请求，统一经过 API client 与 query hook。
- 复杂表单先用受控状态 + 统一校验 helper，不引入额外重表单框架。

**备选方案**：
- 方案 A：Next.js
  - 优点：路由与部署成熟
  - 缺点：当前不需要 SSR，复杂度偏高
- 方案 B：Vue 3
  - 优点：上手快
  - 缺点：与现有参考项目技术基线不一致

## 🔧 后端技术选型

### 技术选择
| 技术 | 选型 | 版本 |
|------|------|------|
| 编程语言 | Python | 3.11.x |
| Web 框架 | FastAPI | 0.115.x |
| ORM | SQLAlchemy | 2.x |
| API 文档 | OpenAPI / Swagger UI | FastAPI 内建 |

### 选型理由
**选择 Python + FastAPI 的原因**：
- 已验证的 CVE graph run 和安全公告处理链的有效行为都来自 Python 生态，认知切换成本最低。
- FastAPI 适合快速定义清晰的场景 API 和平台 API。
- Python 更适合文档抓取、HTML 处理、模型调用和异步任务编排。

**选择 SQLAlchemy 2.x 的原因**：
- 适合在“平台表 + 场景表”边界明确的情况下建立清晰模型。
- 能兼顾 ORM 与显式 SQL 的控制力。

**备选方案**：
- Django：管理后台强，但不适合当前“平台 API + worker + 场景模块”拆分。
- Node.js/NestJS：接口能力足够，但与现有有效能力基线偏离较大。

## 🤖 AI 与 Agent 技术选型

### 技术选择

- **有状态编排框架**：`LangGraph`
- **模型与工具抽象层**：`LangChain` + `langchain-openai`
- **观测与调试**：`LangSmith`
- **运行模式**：优先支持 `graph run`、受限 `Tool-use`、异步后台执行
- **能力预留**：`RAG`、`Multi-Agent`、`Human-in-the-loop`、`MCP`

### 选型理由

**选择 `LangGraph` 的原因**：

- 适合承载 `CVE` 场景这种带循环、分支、停止条件和中间态的显式主链。
- 能把 `graph run` 的节点和状态写清楚，避免把新主线塞回旧 `pipeline/stages` 叙事。
- 便于后续引入 `checkpoint`、中间态恢复和人工接管。

**选择 `LangChain` 的原因**：

- 更适合承担模型封装、结构化输出、工具绑定和提示模板。
- 可以把模型调用与业务编排解耦，不让场景服务直接耦合单一模型 SDK。

**选择 `LangSmith` 的原因**：

- 适合做 `run trace`、调试和评估。
- 可以作为可选观测平面接入，而不是强制运行前提。

### 边界约束

- `LangGraph` 是默认编排框架，不等于所有场景都必须从第一天起做成复杂 graph。
- `LangChain` 只承担模型与工具抽象，不承担业务主编排。
- `RAG`、`Multi-Agent`、`MCP` 等能力进入统一目标架构，但当前 v1 不要求全部同时落地。
- 任何 Agent 能力都不能把系统放宽成开放式网页代理或无边界自动执行平台。

## 🗄️ 数据库选型

### 关系型数据库
**选型**：PostgreSQL

**选择理由**：
- 原生支持 JSONB，适合存放运行摘要、证据片段、规则命中等半结构化数据。
- 单数据库即可承载任务、投递、CVE 结果和公告结果，便于事务和查询统一管理。
- 明确取消 SQLite 双支持，减少实现与测试矩阵。

**版本**：16.x

### 缓存数据库
**选型**：v1 暂不引入

**选择理由**：
- 当前优先解决真实闭环与结构边界问题，不提前引入 Redis 等组件。
- 运行态缓存可以先由前端轮询 + 后端短时内存缓存承担。

### 文件存储
**选型**：本地文件存储

**选择理由**：
- patch、原文快照、抓取 HTML、文本归档更适合先以文件方式保存。
- 后续可无缝切换到 S3 兼容对象存储，但 v1 不提前复杂化。

## 🔄 技术栈对比

### 后端数据库方案对比

| 维度 | PostgreSQL Only | SQLite + PostgreSQL | 选择 |
|------|-----------------|---------------------|------|
| 实现复杂度 | ⭐⭐⭐⭐ | ⭐⭐ | PostgreSQL Only |
| 真实场景承载 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | PostgreSQL Only |
| 测试矩阵 | ⭐⭐⭐⭐ | ⭐⭐ | PostgreSQL Only |
| 历史包袱 | ⭐⭐⭐⭐⭐ | ⭐⭐ | PostgreSQL Only |
| **综合评分** | **✅ 推荐** | - | **PostgreSQL Only** |

**最终选择**：PostgreSQL Only

**理由**：当前仓库从零开始建设，最重要的是稳定建立平台边界和场景闭环，不为兼容历史工程形态增加额外复杂度。

## 📝 技术债务

### 已知问题
- v1 不引入统一任务队列中间件，Worker 扩容策略需要在实现阶段进一步验证。
- 微信/企业微信能力在不同部署环境下可用性不同，需要在运维文档中补充渠道约束。

### 未来优化方向
- 引入对象存储替换本地 Artifact 存储
- 引入专门的队列/调度基础设施
- 按真实需求评估认证、多租户和更细权限
