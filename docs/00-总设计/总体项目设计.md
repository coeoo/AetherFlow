## 💡 核心定义

AetherFlow 的核心定义不是“下游运营平台”，而是一个把原始安全信号加工为结构化情报的情报处理引擎。

最小表达式如下：

```text
原始信号
  -> 采集与归一化
  -> 推理/提取/验证
  -> 结构化情报
```

这里的“原始信号”包括但不限于：

- 网页
- API 响应
- RSS
- 文本正文
- 补丁、附件或页面快照

这里的“结构化情报”不是一个强制统一的数据库表，而是一个概念层结果：

- 在 `CVE` 场景中，表现为 `graph run + fix family + patch evidence`
- 在“安全公告”场景中，表现为 `公告文档 -> 结构化情报包`

本系统的核心目标止于“把原始信号处理成可复查的结构化情报”。  
资产匹配、资产影响计算、终端处置编排、外部运营面板等能力默认属于下游平台，而不是情报理解内核本身。

## 🧱 双层架构

当前仓库采用“双层架构”表达系统边界。

### 第一层：情报处理引擎内核

负责：

- 接收原始信号
- 把大内容沉淀为 `Artifact`
- 进行推理、抽取、候选选择、证据绑定和结果收敛
- 输出场景拥有的结构化结果

这一层是系统的核心定义。

### 第二层：当前 v1 平台壳

负责：

- 任务创建、后台执行、重试与调度
- `Artifact` 管理与读取
- 场景工作台与结果页
- 监控批次、投递记录和系统状态

这一层是当前仓库的承载形态，不等于最终产品边界。

### 两层关系

```text
用户 / 调度器 / API
  -> v1 平台壳
     - Task / Scheduler / Artifact / Workbench / Delivery
  -> 情报处理引擎内核
     - Signal Ingestion / Normalize / Reason / Validate / Emit
  -> 场景结果
     - CVE 结果
     - 公告情报包
  -> 下游平台或人工消费
```

约束说明：

- “平台”只表示当前仓库里的共用底座，不表示产品化已经成立。
- 场景是一等公民，各自拥有独立后端落点、API 与结果模型。
- 平台共享任务、调度、抓取、投递、`Artifact` 与首页壳。
- 平台不共享 `CVE` 与“安全公告”的领域 `run/result schema`。

## 🔄 统一运行时抽象

下面这些对象是总设计层抽象，不要求在数据库层做成一张统一表。

### `Signal`

表示进入系统的原始输入。

示例：

- CVE ID
- 公告 URL
- 正文文本
- RSS 条目
- 第三方 API 返回体

### `Artifact`

表示所有需要复查的大内容与原始材料，是 canonical source。

典型内容：

- HTML
- 归一化正文
- patch / diff / debdiff
- 抓取快照
- 原始附件

### `Evidence`

表示“某个结论为什么成立”的可追溯证据。

要求：

- 能回到 `Artifact`
- 能定位原文片段、规则命中或页面链路
- 能支持人工复核

### `SceneRuntime`

表示某个场景的处理主链。

约束：

- 场景可以共享平台底座
- 场景不能被统一成同一个业务 `run/result schema`

### `SceneRun`

表示某个场景的一次实际执行。

约束：

- 每次场景执行都绑定一个唯一平台 `task_job`
- 平台级 `/retry` 只重跑同一个 `run`
- 全新业务执行必须重新创建新的 `task_job + run`

### `Structured Intelligence Output`

表示场景拥有的结构化输出对象。

说明：

- 这是概念层，不强制统一数据库结构
- `CVE` 与“安全公告”分别定义自己的持久化与接口返回

### `ToolAdapter`

表示引擎调用外部能力的统一适配边界。

示例：

- 页面抓取器
- 数据源解析器
- 模型调用器
- 下载器
- 校验器

### `CheckpointState`

表示长链路执行时的可恢复状态。

作用：

- 支持断点续跑
- 支持中间态调试
- 支持高风险节点的人机接管

### `HumanGate`

表示允许人工审核、确认或放行的关键节点。

这类节点只用于高风险判断，不应把全链路重新写回旧审核台叙事。

## 🧠 技术链分层

本文采用“统一目标架构”写法：下面这些技术都属于 AetherFlow 目标体系的一部分，但并不意味着当前新仓已经全部有代码落地。

### `LangGraph`

定位：

- 默认的 stateful orchestration runtime
- 负责循环、分支、并发、checkpoint 与节点级状态流转

系统职责：

- 显式表达场景主链，而不是把主线塞回旧 `pipeline/stages`
- 让 `CVE` 这类需要 frontier 探索的场景可解释、可恢复、可追踪

边界：

- 不负责模型提供方选择
- 不负责业务表结构设计
- 不把整个系统硬写成“只有 graph 才能运行”

### `LangChain`

定位：

- 默认的 LLM 与工具调用抽象层

系统职责：

- 模型调用封装
- 工具绑定
- 提示模板
- 结构化输出

边界：

- 不承担业务编排主责
- 不把系统变成开放式网页代理

### `LangSmith`

定位：

- 观测、调试、评估与回放平面

系统职责：

- 记录 `run trace`
- 辅助调试和评估
- 在需要时回填外部追踪链接

边界：

- 不是运行前提
- 不应让业务逻辑依赖观测平台可用性

### `RAG`

定位：

- 检索增强模式

系统职责：

- 先检索知识或历史材料，再进入生成、总结或抽取
- 为公告类场景的背景补全、历史比对或同类公告检索预留能力

边界：

- 不是所有场景默认必开
- 不要求 v1 一开始就引入向量召回和索引服务

### `ReAct / Tool-use`

定位：

- 节点内的推理-行动循环模式

系统职责：

- 在有限工具集合内做逐步判断与调用
- 支持“先看结果，再决定抓取/跳转/比对”的局部策略

边界：

- 是运行模式，不是整个系统定义
- 不能放宽到任意网页漫游

### `Multi-Agent`

定位：

- 复杂场景下的高级组合模式

系统职责：

- 在确实有收益时，把采集、抽取、验证、总结等职责拆给专用 agent

边界：

- 不作为当前 v1 默认组织模式
- 不允许为了“看起来先进”而把简单场景过度 agent 化

### `Checkpoint / Memory`

定位：

- 运行时能力，不是业务结果层

系统职责：

- 保存执行状态
- 支持断点续跑
- 支持长任务恢复
- 支持人工接管前后的状态连续性

边界：

- 不能替代业务数据库
- 不能和场景输出表混为一谈

### `Human-in-the-loop`

定位：

- 治理能力

系统职责：

- 在低置信度或高风险节点引入人工审核
- 防止 Agent 在关键结论上误判

边界：

- 只挂关键节点
- 不复活旧 `review / diagnostics` 主叙事

### `Streaming / Async`

定位：

- 执行与交付语义

系统职责：

- 长任务后台异步执行
- 中间态可流式输出或轮询读取
- 结果不阻塞在请求内一次性完成

边界：

- 流式展示是观察方式，不是业务模型本身

### `MCP 协议`

定位：

- 工具与上下文接入协议层

系统职责：

- 标准化接入外部工具、上下文与能力源
- 降低工具调用实现和场景逻辑的耦合

边界：

- 不是场景业务逻辑本身
- 不与平台 API、数据库 schema 混写

## 🎯 场景落点

### `CVE` 场景

当前定位：

- 第一条真实路径
- 目标主线是 `graph run`
- 默认由 `LangGraph` 承载显式状态图

核心链路：

```text
CVE ID
  -> resolve_seeds
  -> plan_frontier
  -> fetch_page
  -> analyze_page
  -> download_patches
  -> finalize_run
```

结果表达：

- `cve_runs`
- `cve_fix_families`
- `cve_patch_artifacts`
- `Artifact` 中的 patch / diff 证据

### 安全公告场景

当前定位：

- 第二条真实路径
- 标准处理对象固定为“公告文档 -> 结构化情报包”

核心约束：

- 手动提交和监控抓取只是两种入口，不是两个独立场景
- 监控批次保留在平台/批次层
- 单文档提取 run 保持场景层独立

结果表达：

- `announcement_runs`
- `announcement_documents`
- `announcement_intelligence_packages`
- 指向原文与归一化正文的 `Artifact`

技术路线：

- 可以先以单文档提取链落地
- 再逐步挂接更强的 `RAG / Tool-use / HITL / Multi-Agent`
- 不要求公告场景一开始就和 `CVE` 场景拥有同等复杂度的 graph

## 🔁 数据与控制流

两个场景都应服从下面这条总链路：

```text
Signal In
  -> Ingestion
  -> Normalize
  -> Artifact 落点
  -> Reason / Extract / Validate
  -> Structured Intelligence Out
  -> Optional Delivery / Downstream
```

其中：

- `Signal In` 进入的是原始信号
- `Artifact` 承担大内容 canonical source
- `Reason / Extract / Validate` 承担场景智能核心
- `Structured Intelligence Out` 才是场景真正交付给下游或页面的对象

当前 v1 平台壳则在这条链路外包一层：

- 创建 `task_job`
- 记录 `task_attempt`
- 绑定场景 `run`
- 暴露工作台与结果页
- 在需要时做投递
