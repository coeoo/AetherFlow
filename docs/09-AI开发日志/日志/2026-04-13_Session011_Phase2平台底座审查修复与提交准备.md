## 关键决策

- `artifact_root` 统一在 `Settings` 构造阶段解析为基于 worktree 根目录的稳定绝对路径，不再让 runtime 代码依赖进程当前工作目录。
- `create_engine_from_url()` 与 `create_session_factory()` 以 `database_url` 为键缓存复用，`health_summary` 不再在每次请求后显式 `dispose()` 缓存 engine。
- `write_text_artifact()` 的数据库持久化阶段被提取为 `_persist_artifact_records()`；如果数据库事务失败，必须回删已写文件和空目录。
- Worker 的失败补偿顺序固定为：先 `mark_attempt_failed()`，再 best-effort `delete_artifact()`；如果清理再失败，只把错误附加到原始异常 note，不覆盖原始失败原因。
- 使用同一个 `TEST_DATABASE_URL` 时，凡是会重建 `public schema` 的 PostgreSQL 测试必须串行运行，不能并发跑多个 pytest 命令。

## 问题与处理

- 问题：默认 `artifact_root` 使用相对路径时，会随着 runtime 启动目录不同而解析到不同位置。
- 处理：把默认值和手工传入值都统一解析成稳定绝对路径，并补路径稳定性测试。

- 问题：Artifact 文件先写盘、后入库，数据库持久化失败会遗留孤儿文件。
- 处理：增加 `_persist_artifact_records()` 持久化封装，并在失败时清理文件与空目录。

- 问题：Worker 在 `mark_attempt_succeeded()` 失败后的清理路径中，如果 `delete_artifact()` 再失败，会覆盖原始异常。
- 处理：先落失败状态，再执行 best-effort 清理，清理异常只附加到 `RuntimeError.__notes__`。

- 问题：一次把两个共享同一 PostgreSQL 库的 pytest 命令并行跑，会因为 `prepare_database()` 重建 `public schema` 而互相踩库。
- 处理：最终验证证据只保留串行运行结果，并把这条验证约束写回文档。
