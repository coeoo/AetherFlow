## 做了什么

- 先基于真实浏览器联调复盘了三个样本：
  - `CVE-2024-3094` 会被单个失效页面拖成 `fetch_failed`
  - `CVE-2024-6387` seed 中已有 direct commit，但被页面预算截断后失败为 `no_patch_candidates`
  - `CVE-2024-21626` 因 direct commit 排位靠前而成功
- 把下一阶段收口为 **Phase B2：主链鲁棒性增强**
- 按 TDD 补失败测试，覆盖：
  - seed direct candidate 优先消费
  - 页面抓取局部容错
  - 工作台 active run 终态后历史刷新
- 实现后端主链增强：
  - `plan_frontier` 跳过 direct patch/commit 引用
  - runtime 先构造 direct seed candidates，再抓普通页面
  - 页面抓取改为逐页容错
- 实现前端状态同步：
  - active run 进入终态后主动刷新最近运行列表
- 回写本轮相关文档、设计规格、实施计划和使用说明

## 关键决策

- 继续坚持 **实时获取**，不引入离线 `cvelistV5` 基线
- 本轮不引入 graph runtime、LLM fallback 或 family 持久化模型
- 先解决高信号被页面预算截断的问题，再继续评估更复杂演进
- 页面抓取失败改为局部失败，只有在无 direct candidate 且无成功页面时才收口为 `fetch_failed`
- 工作台最近运行列表只在 active run 终态时刷新一次，不做长期高频轮询
