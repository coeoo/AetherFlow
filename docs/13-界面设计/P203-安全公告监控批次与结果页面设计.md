# P203 安全公告监控批次与结果页面设计

> **对应模块：M203 安全公告调度运行与结果**

---

## 🎯 页面目标

`/announcements?tab=monitoring` 是公告场景的批次视图，负责把“监控源抓取批次”和“单文档提取 run”两个层级分开展示。

页面必须让用户一眼区分：

1. 这次批次抓了多少条。
2. 哪些是新增文档。
3. 为哪些新增文档创建了 `announcement_run`。
4. 某条 run 最终是否进入情报包详情。

---

## 🚪 入口与出口

### 入口

- `P201` 点击 `查看监控批次`
- `P202` 试跑完成后跳转
- 直接访问 `/announcements?tab=monitoring`

### 出口

- 点击某个 run -> `/announcements/runs/{run_id}`
- 点击 `监控源管理` -> `/announcements/sources`
- 点击 `返回手动提取` -> `/announcements`

---

## 🧱 页面布局

```text
Topbar
  -> Monitoring Hero
  -> Batch List
  -> Batch Detail Panel
  -> Triggered Runs List
```

### 区块1：Monitoring Hero

- 当前页面标题
- 简短说明：查看抓取批次、新增文档和提取联动
- 快捷入口：返回手动提取、进入源管理

### 区块2：批次列表

- 批次按时间倒序
- 每条批次显示：
  - `fetch_id`
  - `source_id`
  - 源名称/类型
  - 状态
  - 发现条目数
  - 新增条目数
  - 触发 run 数量

### 区块3：批次详情面板

- 选中一条批次后，在页内右侧或下方展开：
  - 批次状态
  - 时间窗口
  - 去重摘要
  - 失败条目摘要

### 区块4：Triggered Runs List

- 展示 `trigger_fetch_id = fetch_id` 的所有 run
- 每条 run 显示：
  - run 状态
  - 文档标题
  - 结果摘要
  - 进入情报包详情的动作

---

## 🖱️ 关键交互

- 页面默认先展示批次列表；详情面板只在选中批次后展开。
- 批次与 run 不在同一层级混排，避免误把抓取批次当成单篇公告结果。
- 点击某个 run 直接进入情报包详情，不新增“run 管理后台”。
- 试跑刚结束时，页面应自动高亮本次 `fetch_id`。

---

## 🎭 状态稿

### 默认态

- 展示最近批次列表，最新批次默认选中。

### 空态

- 尚无批次时提示：`当前还没有监控抓取记录，请先创建源或立即试跑`。

### 部分失败态

- 批次成功但部分文档提取失败：
  - 批次行显示成功
  - 详情面板显示失败文档数与失败原因摘要

### 批次失败态

- 批次未抓到数据或源异常：
  - 列表中显示失败状态
  - 详情面板提供失败原因

### 去重提示态

- 对命中 `content_dedup_hash` 的新文档，只标记“可能重复”，不阻断 run 展示。

---

## 📦 页面视图对象

### `MonitorBatchSummary`

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `fetch_id` | string | 批次 ID |
| `source_id` | string | 源 ID |
| `source_name` | string | 源名称 |
| `status` | string | 批次状态 |
| `discovered_count` | number | 发现条目数 |
| `new_count` | number | 新增数 |
| `extraction_run_count` | number | 提取 run 数 |

### `MonitorBatchDetailView`

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `fetch_id` | string | 当前批次 |
| `dedup_summary` | object | 去重摘要 |
| `failed_items` | array | 失败条目摘要 |
| `triggered_runs` | array | 关联的提取 run 列表 |

---

## 🔌 API 与字段映射

| 页面区块 | API | 主要字段 |
|----------|-----|----------|
| 批次列表 | `GET /api/v1/announcements/monitor-runs` | `fetch_id`、`source_id`、计数字段 |
| 批次详情 + run 列表 | `GET /api/v1/announcements/monitor-runs/{fetch_id}` | 批次摘要、`triggered_runs` |

约束：

- `triggered_runs` 的关联方式固定为 `announcement_runs.trigger_fetch_id = fetch_id`。
- 批次对象必须能直接返回 `source_id`，不通过弱语义猜测。
