## 🎯 功能目标

### 业务目标
把首批三类真实来源的抓取差异固化为独立适配器，使监控链路既统一，又不丢失来源特性。

### 用户价值
- 监控结果不是“泛化 RSS 抓取”，而是基于各来源真实特点实现。
- 每类源出现异常时，可以精确定位到适配器层而不是把整个场景搞混。

---

## 👥 使用场景

### 场景1：微信公众号/微信文章源
**场景描述**：复用已验证的登录态和文章抓取逻辑，拉取目标账号最新文章。

### 场景2：Openwall
**场景描述**：按日期页回溯列表，过滤回复和重复标题，抓取新消息。

### 场景3：NCC Security Community
**场景描述**：通过动态页面抓取漏洞列表，识别新增条目。

---

## 🔄 业务流程

### 主流程
```text
监控源触发
  -> 根据 source_type 选择适配器
  -> 执行列表抓取
  -> 输出标准化文档条目
  -> 交给 M203 继续去重和提取
```

## 💾 数据设计

### 涉及的数据表
- `announcement_sources`
- `source_fetch_records`
- `announcement_documents`

### 核心数据字段

#### StandardSourceDocument
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| source_name | string | 是 | 来源名 |
| source_type | string | 是 | 来源类型 |
| title | string | 是 | 标题 |
| source_url | string | 是 | 原始链接 |
| published_at | string | 否 | 发布时间 |
| source_item_key | string | 是 | 来源内稳定唯一键 |
| source_artifact_ref | string | 否 | 如果适配器已抓到正文，则返回 Artifact 引用而不是直接返回大文本 |
| content_dedup_hash | string | 否 | 内容指纹，用于跨来源重复提示 |

## 🔌 接口设计

### 接口1：无直接用户接口
来源适配器由监控任务内部调用，不单独暴露公共 API。

**业务规则**：
- 适配器输出统一文档条目结构
- 上层不感知来源内部抓取细节

---

## ✅ 业务规则

### 规则1：适配器负责抓取，不负责结构化提取
**规则描述**：来源适配器只输出标准文档条目，不直接生成情报包。

### 规则2：不同源使用不同去重键策略
**规则描述**：
- 微信：文章 URL 或平台可稳定识别的文章 ID
- Openwall：消息 URL
- NCC：漏洞 ID 或详情 URL

这些键统一映射为 `source_item_key`，并在数据库中以 `(source_id, source_item_key)` 做源内幂等。

### 规则3：动态抓取只在必要来源使用
**规则描述**：Playwright 只用于 NCC 等确实需要动态页面解析的来源。

### 规则4：适配器不返回 canonical 大文本
**规则描述**：如果适配器阶段已经拿到正文，必须先交给 M004 落 Artifact，再把 Artifact 引用传给上层，不直接把大文本作为持久化契约。

## 🚨 异常处理

### 异常1：微信登录态失效
**触发条件**：存储态过期或被踢下线

**错误提示**：源试跑失败，提示登录态无效

**处理方案**：标记失败并提示重新配置登录态

---

### 异常2：Openwall 页面结构变更
**触发条件**：日期页或消息页解析失败

**错误提示**：抓取失败

**处理方案**：保留响应快照与失败记录，便于修适配器

---

### 异常3：NCC 动态加载失败
**触发条件**：页面渲染超时或 DOM 选择器失效

**错误提示**：NCC 源抓取失败

**处理方案**：保留 Playwright 抓取日志与 HTML 快照

## 📝 开发要点

### 技术难点
1. 三类来源抓取模式差异很大，必须保持适配器边界清晰。
2. 要把 `SecurityAnnouncement` 的已验证行为迁移为“行为参考”，而不是直接搬旧类结构。

### 性能要求
- Openwall 和微信抓取要控制最大回溯窗口
- NCC Playwright 抓取要设置更严格的页面超时

### 注意事项
- 适配器只做获取与标准化
- 通知和提取逻辑必须放到上层模块
