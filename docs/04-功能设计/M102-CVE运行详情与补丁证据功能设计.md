## 🎯 功能目标

### 业务目标
提供一个“可解释”的运行详情页，完整展示一次 CVE graph run 的结论、证据链、fix family、patch 和 diff。

### 用户价值
- 用户不仅知道有没有补丁，还知道系统是如何找到补丁的。
- 可以在详情页里直接完成复核，而不必回外部站点逐一打开。

---

## 👥 使用场景

### 场景1：复核主补丁
**场景描述**：用户想确认系统标记的主补丁是否可信。

**用户操作流程**：
1. 打开 `/cve/runs/{run_id}`
2. 查看主结论卡片
3. 查看 primary family 和主 patch
4. 打开 diff 内容核查

---

### 场景2：查看页面探索过程
**场景描述**：用户想知道系统为什么从某个公告页跳到某个 patch 链接。

**用户操作流程**：
1. 在详情页查看 trace timeline
2. 展开步骤明细
3. 查看 selection_reason、matched_rule_ids 和 stop_reason

---

## 🔄 业务流程

### 主流程
```text
进入 run 详情
  -> 读取 run 完整数据
  -> 展示结论卡片
  -> 展示 fix family / patch 列表
  -> 展示 trace 时间线
  -> 按需加载 diff 内容
```

## 🎨 界面设计

### 页面1：CVE 运行详情页
**页面路径**：`/cve/runs/:runId`

**页面元素**：
- 顶部结论卡片

## 💾 数据设计

### 涉及的数据表
- `cve_runs`
- `cve_fix_families`
- `cve_patch_artifacts`
- `artifacts`

### 核心数据字段

#### CVERunDetail
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| run_id | string | 是 | 运行 ID |
| cve_id | string | 是 | CVE 编号 |
| status | string | 是 | 状态 |
| stop_reason | string | 否 | 停止原因 |
| summary | object | 是 | 运行摘要 |
| fix_families | array | 是 | 补丁家族 |
| patches | array | 是 | 补丁记录 |
| source_traces | array | 是 | 页面探索证据 |

## 🔌 接口设计

### 接口1：获取运行详情
**接口路径**：`GET /api/v1/cve/runs/{run_id}`

### 接口2：获取补丁列表
**接口路径**：`GET /api/v1/cve/runs/{run_id}/patches`

### 接口3：获取补丁内容
**接口路径**：`GET /api/v1/cve/runs/{run_id}/patch-content?candidate_url=...`

**业务规则**：
- 补丁内容按需加载
- 如果 Artifact 不存在，返回 404

---

## 📦 前端状态对象

#### PatchDiffPanelState
| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| candidate_url | string | 否 | 当前查看的补丁 URL |
| loading | boolean | 是 | 是否正在加载 diff |
| content_available | boolean | 是 | 是否存在 diff 内容 |
| error_message | string | 否 | diff 加载失败提示 |

---

## 🔁 子流程/状态机

### 详情页与 Diff 查看状态机
```text
detail_loading
  -> detail_ready
  -> detail_empty

detail_ready
  -> diff_idle
  -> diff_loading
  -> diff_ready
  -> diff_failed
```

**状态说明**：
- 详情页加载与 diff 加载分成两个状态机，避免局部失败拖垮整页。
- `detail_empty` 用于无效 `run_id` 或结果不存在场景。

## ✅ 业务规则

### 规则1：先给结论，再给证据
**规则描述**：详情页顶部必须先说明主结论，而不是先显示调试数据。

### 规则2：主补丁由 primary family 决定
**规则描述**：如果存在主家族，则优先展示主家族下的主补丁。

### 规则3：证据链可读性优先
**规则描述**：trace 不只是原始 JSON，要整理为人类可读的时间线与步骤原因。

---

## 🚨 异常处理

### 异常1：run 不存在
**触发条件**：通过无效 run_id 打开详情页

**错误提示**：`运行记录不存在`

**处理方案**：展示空态并允许返回工作台

---

### 异常2：diff 内容缺失
**触发条件**：patch 记录存在，但 Artifact 丢失或下载失败

**错误提示**：`补丁内容不存在或尚未下载`

**处理方案**：保留 patch 元数据，标记内容不可查看

## 📝 开发要点

### 技术难点
1. trace、family、patch 三类信息既有关联又不能混在一个区块里。
2. diff 内容可能很长，需要按需加载和懒渲染。

### 性能要求
- 详情接口响应目标 < 500ms
- diff 内容接口按需加载，避免首页一次返回大文本

### 注意事项
- 详情页是“证据页”，不是“原始 JSON 页”
- 原始字段可以保留为开发辅助，但不能是主视图
