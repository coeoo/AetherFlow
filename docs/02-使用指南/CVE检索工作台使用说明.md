# CVE Patch 搜索工作台使用说明

> **如何使用 CVE Patch Agent 发起一次补丁搜索**

---

## 1. 功能简介

`/cve` 是 CVE Patch Agent 的主入口。
用户输入一个 `CVE ID` 后，系统会执行：

1. 多源 seed 聚合
2. 首轮候选识别
3. 多跳页面探索
4. patch 收敛与下载校验
5. 搜索图与证据结果输出

如果最终命中 patch，页面会给出主结论与详情页入口；
如果未命中，页面仍会保留搜索摘要，并引导进入详情页查看完整搜索图。

---

## 2. 使用步骤

1. 打开页面 `/cve`
2. 在输入框中输入 CVE，例如 `CVE-2024-3094`
3. 点击“开始搜索”
4. 观察运行状态、最近动作与预算推进
5. 查看结果摘要，判断是否已经找到主 patch
6. 点击“查看详情”进入 `/cve/runs/{run_id}`，继续复核搜索图、
   决策历史、候选收敛和 diff
7. 如需回看历史搜索，可在“最近运行”区域直接进入对应详情页

---

## 3. 页面说明

- **输入框**：仅接受合法 CVE 编号
- **运行状态卡片**：展示当前阶段、最近动作和停止原因
- **预算摘要**：展示页面预算、深度预算和下载预算的使用情况
- **结果摘要**：展示是否命中 patch、主结论和下一步动作
- **最近运行**：展示最近搜索记录，便于回看不同 run 的结论
- **详情页入口**：进入完整搜索图和证据详情

---

## 4. 详情页里能看到什么

进入 `/cve/runs/{run_id}` 后，可以继续查看：

- 最终 patch 结论
- 从 seed 到 patch 的搜索图
- 当前或最后一轮 frontier
- 决策历史与校验结果
- patch 收敛关系
- diff 内容与原始证据

---

## 5. 常见问题

### 为什么系统还在运行中？

- 说明 Agent 仍在探索页面、收敛候选或尝试下载 patch。
- 工作台只展示搜索摘要；如果想知道正在访问哪些页面，应进入详情页查看搜索图。

### 为什么没有马上找到 patch？

- 真实世界中的 patch 链路常常不是直达链接。
- 系统可能需要先经过公告页、vendor tracker、Bugzilla、代码评审页等
  bridge 页面，再继续逼近 patch。
- 只要预算尚未耗尽，Agent 会继续基于当前 frontier 做下一轮决策。

### 如果没有显式 patch 链接，会直接失败吗？

- 不会。系统会把当前页面中的高价值链接整理为候选 frontier，
  交给 Agent 判断哪些值得继续深入。
- 只有在预算耗尽、候选全部被判定为低价值，或所有路径都无法收敛时，
  才会结束本次搜索。

### 为什么工作台里看不到完整链接树？

- 工作台负责输入、摘要和状态提示，不负责承载完整搜索图阅读。
- 完整图、预算明细和决策历史统一放在详情页。

### 为什么未命中 patch 也建议看详情页？

- 因为“没找到”本身也需要可解释。
- 详情页会告诉你系统访问了哪些页面、为什么继续或剪枝、
  是预算耗尽还是证据不足，从而帮助判断是否需要人工接手。

### 为什么有多个 patch 候选？

- 不同来源可能指向同一个 fix family，也可能分别指向多个 patch 入口。
- 系统会在详情页展示候选如何收敛为主结论，以及每个候选的来源强度。

---

## 6. 结果解释

- **已命中 patch**：系统找到并校验了可用 patch/diff，详情页可继续复核来源链路
- **搜索中**：系统仍在探索页面、扩展 frontier 或下载候选
- **未收敛**：本次运行在当前预算内没有得到足够强的 patch 结论
- **建议查看详情**：表示需要通过搜索图、预算和决策历史进一步判断原因
